# SHL Study Guide

## 这套材料现在怎么用

当前唯一主文档是 [SHL_Final_Full_Problems_Annotated_v2.md](/G:/Documents/JavaProject/java-study/Leetcode-demo/SHL题目参考/SHL_Final_Full_Problems_Annotated_v2.md)。

你现在可以把整套材料分成三层来学：

- 第一层：先只读英文题面，练 `Question / Input / Output / Constraints / Example / Explanation` 的读题速度。
- 第二层：再看每题下面的“校对与理解备注”，理解这题到底是原题恢复、还是学习版定稿。
- 第三层：最后对照 `src/main/java/com/aquarius/wizard/leetcode/shl` 里的 Java 实现，把题意和代码绑定起来复习。

如果一道题被明确标成“学习版定稿”，你就把它当成当前项目里的正式练习版本，不要再纠结“它是不是百分之百等于原截图”。你的目标不是考古，而是建立一套可学习、可验证、可复习的题库。

## 读题时先看什么

不要一上来先看中文备注，也不要一上来就看代码。

先看这三个东西：

1. `Output`
2. `Constraints`
3. `Example`

原因很简单：

- `Output` 决定你到底要打印什么对象。
- `Constraints` 决定你允许用什么复杂度。
- `Example` 决定你当前理解的对象是不是和题目说的是同一个东西。

很多题不是算法难，而是你一开始就把“数的是什么”或“输出的是什么”看错了。

## 这批题最容易读错的地方

### 1. `ways` 到底在数什么

看到 `number of ways`，必须先问自己：

- 在数不同操作方案？
- 在数不同结果？
- 在数不同窗口 / 子串 / 子数组？
- 在数不同值对，还是不同下标对？

`B03` 就是这类坑的典型。

它原文里的 `different ways` 有歧义，而样例又不足以区分多种解释。所以当前主版本直接把它定成“最优窗口数”学习版题面。这个决定本身就是一种很重要的读题训练：当原题不稳时，你要学会先把“计数对象”钉死。

### 2. 输出的是集合，还是顺序

有些题写着“print IDs”，不代表你要输出构造顺序。

你要先分清：

- 题目要的是合法解集合
- 还是某个具体排列 / 路径 / 执行顺序

这类错一旦发生，代码可能连样例都能过，但整体题意已经偏了。

### 3. OCR 最容易坏的不是正文，而是约束

模糊图片里最常坏的是：

- 次方和指数
- 下标范围
- `at most` / `exactly`
- `<= K` / `= K`
- 是否允许重复
- 是否要取模

所以做 OCR 题时，约束和 Note 跟 Question 一样重要。

`B09` 就是典型例子：样例和代码都明确是 0-based，但原约束写法是错的。  
`B08` 也是典型例子：原文里的 `mod 1000007` 放在“最小时间”问题上，逻辑上就很可疑。

### 4. 题面和复杂度不一致时，不要硬信两者都对

如果正文看起来是某个问题，但约束明显不支持这个问题的精确解，你要先怀疑题目材料坏了。

`A25` 就是这样。

按 OCR 字面，它更像“最少直线覆盖点集”；但原约束像是另一个题。当前主版本没有假装“原题已完整恢复”，而是明确把它定成学习版，并把有效规模改成去重点数 `<= 20`，让状态压缩 DP 真正成立。

这类处理方式本身就值得你学会。

## 怎么判断一题该用什么复杂度

做这批题时，先用约束反推复杂度：

- `10^3` 左右：`O(n^2)` 经常还能接受。
- `10^5` 左右：优先想 `O(n log n)` 或 `O(n)`。
- `2*10^5` 左右：滑动窗口、前缀和、堆、图的线性/近线性算法更常见。
- 如果正文像 NP 难问题、但约束却很大，优先怀疑题面不完整或 OCR 出错。

这比死记“这题像什么 LeetCode”更有用。

## 现在这批题里最值得复用的思考方式

### 1. 先问“题目在数什么”

遇到计数题时，先把对象写成一句自己的话：

- 数窗口
- 数结果串
- 数方案集合
- 数值对
- 数路径

只要这一步没写清楚，后面的代码和样例都可能是伪正确。

### 2. 先写 brute force，再写正式解

对来源不稳的题，这个习惯特别值钱。

正确流程应该是：

1. 先写小规模暴力。
2. 再写正式解。
3. 再用随机数据对拍。
4. 最后再决定你当前理解的题意是不是站得住。

这次很多题能定稿，不是因为一开始就猜对了，而是因为后面做了对拍和反证。

### 3. 主动造“区分样例”

原样例往往太弱，不足以区分歧义解释。

你要学会自己造能拆分不同解释的小反例。

例如 `B03`：

- 原样例 `1010101 / 1 -> 3` 太弱，区分不了多种解释。
- 但 `S = 101, K = 0` 能立刻区分“窗口数”和“结果串数”。

这就是为什么学习时不能只依赖题目给的 Example。

### 4. 对优化题里的“取模”保持警惕

如果题目问的是：

- 最小时间
- 最短路径
- 最小成本
- 最优值

那“结果对某个数取模”通常就不自然。

这不是绝对不可能，但必须提高警惕。  
`B08` 这次就是这样收口的：既然题目本质上是最小时间优化题，而取模规则又缺乏自洽性，就直接删掉，明确改成学习版定稿。

## 难度标签怎么用

当前主文档和 Java 类里都已经标了：

- `简单`
- `中等`
- `困难`

这个难度不要只按算法模板理解，更要按“读题建模难度”来理解。

这批 SHL 题里，经常会出现：

- 算法不算最难
- 但题目容易误读

所以你复习时可以额外问自己一句：

`这题真正难，是实现难，还是理解题意难？`

如果答案是后者，那你复习时就不要只看代码，要回看题面和备注。

## 我建议你复习时的固定流程

1. 先只看英文题面，自己口头复述题意。
2. 再看 `Output` 和 `Constraints`，说出你预期的复杂度。
3. 然后看 Example，确认你数的对象没错。
4. 再看“校对与理解备注”，确认这题是原题恢复还是学习版定稿。
5. 最后再看 Java 实现，验证你的建模是否和代码一致。

如果你能稳定做到这 5 步，你后面读类似 OA 题会快很多。

## 最后一个建议

你现在做的不是“把一堆图片题转成文字”，而是在建立自己的题感数据库。

这意味着每次复习时，除了问“这题怎么做”，还应该再问一句：

`如果下次给我一张模糊图片和一个可疑样例，我应该怎么把题目本身读对？`

这批 SHL 题最有价值的部分，就在这里。
